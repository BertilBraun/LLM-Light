# Architecture

LLM-Light is organized around typed experiment configuration, ordered pipeline
stages, local filesystem artifacts, and standalone scripts for dataset
generation and run export. A run should be inspectable and repeatable without
reading every source file. The project is small by budget, but the system was
designed around concepts that scale: synthetic corpus generation, sharded
datasets, parallel preprocessing, packed sequence shards, rank-aware
distributed training, throughput logging, TensorBoard observability, and
exportable run artifacts.

## Pipeline

The current stage order is:

```text
raw_dataset
processed_dataset
tokenizer
packed_dataset
pretraining
post_training
evaluation
```

Each stage receives validated configuration, checks upstream artifacts, writes
its own manifest, and can be skipped when its existing output is compatible.
Recomputation is handled by removing the relevant artifact directory and
rerunning the same plan.

`--from` and `--to` make it possible to run a subset of stages, for example
preparing packed data before a GPU training launch.

## Run Layout

A typical run directory is:

```text
runs/<experiment>/
  resolved_config.json
  pipeline.jsonl
  performance.jsonl
  artifacts/
    raw_dataset/
    processed_dataset/
    tokenizer/
    packed_dataset/
    pretraining/
      manifest.json
      metrics.jsonl
      training_evaluations.jsonl
      tensorboard/
      checkpoints/
    post_training/
    evaluation/
      manifest.json
      report.json
```

Large artifacts stay in `runs/` or external storage. The `export` stage writes
a compact archive containing the config, manifests, tokenizer, metrics,
evaluation reports, TensorBoard logs, and latest checkpoint.

## Configuration

Experiment files are YAML files validated by Pydantic models in
`llm_lite/config/models.py`. The top-level blocks are:

```text
experiment
dataset
preprocessing
tokenizer
packing
model
training
post_training
evaluation
inference
distributed
```

Execution controls such as dry-run, stage ranges, and forced recomputation are
CLI arguments rather than experiment fields.

## Dataset Ingestion

Implemented dataset sources:

- `inline_text`: small text literals for verification and tests.
- `local_text`: explicit paths and glob patterns over UTF-8 text files.
- `huggingface`: streaming or regular Hugging Face datasets with either a
  single text column or a format template over multiple columns.

Hugging Face configs can map source splits into pipeline splits and can skip a
prefix of documents when several pipeline splits are carved from one source
split.

Example:

```yaml
dataset:
  type: huggingface
  name: BertilBraun/TinyPython
  text_template: "{task_description}\n\n{code}\n"
  streaming: true
  splits:
    - source_split: train
      split: validation
      max_documents: 1000
    - source_split: train
      split: test
      skip_documents: 1000
      max_documents: 1000
    - source_split: train
      split: train
      skip_documents: 2000
```

## Synthetic Dataset Generation

The TinyPython training corpus was generated by this project and uploaded to
Hugging Face as `BertilBraun/TinyPython`.

The generator lives in:

```text
llm_lite/scripts/generate_tinypython.py
```

It implements:

- A semantic seed space over input types, task families, operations, conditions,
  edge behaviors, implementation styles, naming styles, and task details.
- Prompt rendering for local instruction-tuned teacher models through vLLM.
- Stochastic multi-sample generation per seed.
- Parsing of `<task>` and `<code>` blocks.
- Python AST validation and single-top-level-function enforcement.
- Signature, normalized description, family, tag, and seed metadata recording.
- Invalid-generation JSONL output with rejection reasons.
- Resume logic based on completed seed/sample attempts.

This generation path is separate from model pretraining. The uploaded public
dataset is then consumed through the normal Hugging Face ingestion path.

## Preprocessing

Implemented transforms:

- Unicode normalization.
- Line-ending normalization.
- Lowercasing.
- Minimum and maximum character length filters.
- Exact deduplication.
- Deterministic split assignment.
- Python function extraction.

Preprocessing writes split-sharded text artifacts and aggregate statistics.
Parallel preprocessing is available through the `preprocessing.workers` config
field. The current implementation runs local worker processes, but the artifact
shape is intentionally compatible with a larger external scheduler or
distributed preprocessing system.

When the source dataset already provides split membership, do not configure
`assign_split`.

## Tokenizers

Implemented tokenizers:

- `character`: simple verification tokenizer.
- `byte_bpe`: Python byte-level BPE implementation.
- `rust_byte_bpe`: tokenizer backed by the `tokenizers` package.

Byte-level BPE starts from bytes plus special tokens and trains deterministic
merges up to the configured vocabulary size. Training uses a bounded prefix
sample controlled by `max_training_documents` and/or `max_training_bytes`.

The central tokenizer invariant is:

```python
tokenizer.decode(tokenizer.encode(text)) == text
```

This matters for Unicode, whitespace, tabs, newlines, Python indentation, and
string syntax.

## Packing and Data Loading

The packing stage creates fixed-length autoregressive sequences and writes
compact shard files plus indexes. It supports BOS/EOS insertion,
cross-document packing, configurable context length, maximum shard token count,
and parallel workers.

Training uses packed sequence datasets rather than re-tokenizing raw text during
optimization. Packed data can be benchmarked with:

```bash
python -m llm_lite.scripts.benchmark_packed_datasets
```

## Models

Implemented model families:

- `dense_gpt`: decoder-only Transformer.
- `moe_gpt`: decoder-only Transformer with MoE feed-forward blocks.

The MoE path supports top-k routing, expert feed-forward layers, auxiliary
router loss, total parameter reporting, and active-parameter reporting.

## Training and Checkpointing

Training currently implements causal language modeling with AdamW-style
configuration, precision selection, gradient clipping, structured metrics,
TensorBoard logging, checkpoint intervals, and checkpoint resume.

Distributed training is launched with `torchrun`. Implemented and validated
distributed execution is data parallel. The configuration model also describes
topology dimensions for tensor, pipeline, context, and expert parallelism, but
values greater than 1 for those dimensions are rejected because they are not yet
implemented.

Checkpoint modes:

- `full`: single-process style checkpoint.
- `sharded`: rank-local distributed checkpoint layout with per-rank state,
  completion markers, a global manifest, latest-checkpoint pointer, validation,
  and resume support.

Compatible `maximum_steps` increases resume from the latest checkpoint.
Architecture, tokenizer, packed data, optimizer shape, batch, precision, and
gradient-clipping changes invalidate pretraining.

## Inference

Implemented inference engines:

- `naive`: full-sequence reference decoding.
- `kv_cache`: autoregressive generation with cache reuse.

Implemented decoding modes:

- greedy decoding
- top-k sampling with temperature

The generation CLI loads the tokenizer and latest pretraining checkpoint from a
completed run:

```bash
python -m llm_lite.scripts.generate \
  --config configs/python_moe_full.yaml \
  --prompt "Given a string, write a function that returns the string in reverse order." \
  --maximum-new-tokens 160
```

## Evaluation

Implemented evaluators:

- `exact_reproduction`: verifies tiny memorization runs.
- `perplexity`: evaluates validation loss and perplexity.
- `fixed_prompt_generation`: records generation samples for configured prompts.
- `python_completion`: generates Python continuations, parses source, runs
  checks in a subprocess, and reports parse, execution, and check-pass metrics.

The Python completion evaluator trims trailing newlines from signature prompts
before generation. Evaluation metrics can run at the end of a pipeline or during
training through `training.evaluation`.

Training-time evaluation metrics are written to JSONL and TensorBoard.

## Post-Training

Implemented post-training configuration paths include:

- `none`
- direct preference optimization from a preference JSONL dataset
- generated Python DPO data from task prompts and execution feedback

This is implemented as project infrastructure. The latest reported TinyPython
MoE result is a base pretraining run, not a post-trained model.

## Known Limits

- Current models are small and should not be described as broadly capable code
  models.
- Data parallelism is implemented; tensor, pipeline, context, and expert
  parallelism are represented in config but not implemented.
- Fully sharded data parallel exists as configuration/runtime direction, but it
  is not a validated result in this repository.
- Dense-versus-MoE experiment comparison is still future work.
- The Python MoE result plateaued on the current completion benchmark, so more
  progress likely requires better data coverage or evaluation-shaped training
  examples rather than simply extending the same run.
